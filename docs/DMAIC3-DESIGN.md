# DMAIC3 設計策略與布局

> 醫療器材失效調查系統 — 單檔 `DMAIC3.html` 的架構、寫作策略、版面與設計決策。
> 目的:讓後續任何人(或 AI)接手修改時,能理解「為什麼這樣寫」,而不只是「寫了什麼」。

---

## 1. 專案定位(不可動搖的前提)

- 本系統**不做臨床決策、不診斷、不取代醫護**。它輔助 QA / 製造 / 製程 / R&D / RA 進行失效與 CAPA 調查。
- **最終目標是降低產品風險,不是產生報告。** 報告只是最後交付物;每一個產生的東西都必須服務於「降低風險」。
- 最終輸出**一律由合格工程與 QA 人員審閱**,因此需要工程級推理,而非輕量對話式回應。
- 設計哲學(貫穿全檔):
  - **Context Preservation** — 避免越做越偏;每階段輸出寫入 `STATE`,後續只從 `STATE` 讀。
  - **Traceability** — 所有結論可追溯來源(問卷 / AI 調查 / 證據庫)。
  - **No Hallucination** — 不知道就標「待釐清」,絕不臆測數據、標準條號或結論。
  - **Engineering First** — 工程優先,不寫漂亮空話。
  - 每個結論附**信心度 + 證據來源**;每個驗證項對應**假設 ↔ 失效模式 ↔ 風險**。

---

## 2. 工作流(八階段,單向資料流)

規格要求「每個階段都必須影響下一階段,沒有一節能獨立存在」。實作方式:**單一 `STATE` 物件**,每階段把輸出寫進去,下一階段只讀 `STATE`,不重算上游。

```
1 問題定義   device / problem / invType
        ↓ 寫入 STATE.device, STATE.problem
2 動態問卷   分支式選項(答案改變後續題目與因子預選)
        ↓ 寫入 STATE.questionnaire.{answers,flags}
3 AI 互動調查 逐步收斂,回答標記到魚骨因子
        ↓ 寫入 STATE.investigation.{turns,factorHints,summary}
4 證據彙整   內部(問卷/AI)+ 外部(510k / MAUDE / ISO 示範庫)
        ↓ 寫入 STATE.evidence[]
5 魚骨 / 根因 8 因子勾選 → 信心度 → 根因清單
        ↓ 寫入 STATE.factors[], STATE.rootCauses[]
6 風險排序   FMEA:RPN = S × O × D(預設值由證據/信心度推估)
        ↓ 寫入 rc.fmea,重排 rootCauses
7 驗證計畫   依 RPN 由高到低;每項說明為何需要/驗證哪個假設/消除哪個失效/降哪個風險
        ↓ 寫入 STATE.verifications[]
8 最終報告   可追溯結論 + 匯出 JSON
        ↓ 寫入 STATE.report
```

**關鍵不變式:** 階段 N 只能讀 `STATE`(來自 N-1 以前),不得重新向使用者索取上游已收集的資料。這就是 Context Preservation 的具體落地。

## 3. 雙軌推理:規則引擎(保證可追溯) + AI 連線(工程級深度)

這是 DMAIC3 相對前身最重要的架構決策。

| 產出 | 規則引擎(離線,永遠先算好) | AI 連線模式(有後端時覆蓋敘述) |
|---|---|---|
| 問卷追問、因子預選、信心度分數、RPN | ✅ 全部由規則決定 | ❌ 不交給模型(保證可重現、可稽核) |
| 調查摘要(階段 3) | 規則版摘要 | 模型產生 150–220 字收斂摘要 |
| 根因工程佐證與缺口(階段 5) | 規則版 support/missing | 模型補 rationale/gap(**只加敘述,不改結構或信心度**) |
| 報告結論(階段 8) | 規則版結論 | 模型產生可追溯結論段落 |

設計理由:**信心度、風險分數、因子結構必須可重現、可稽核**,所以由規則引擎決定,模型不得竄改;模型只在「工程敘述」層加值。這樣即使模型不可用(離線),整條流程仍完整可跑,且結論的量化基礎不變。

- 連線偵測:`pingAI()` 對 `POST /api/dmaic {task:'ping'}` 探測;`file://` 直接開啟時略過(避免 CORS 雜訊)。
- 徽章:`🟢 AI 連線模式` / `🟡 模擬模式`。
- `aiCall(task, context, opts)`:失敗一律回 `null` → 前端自動退回規則版。**模型不可用絕不讓流程壞掉。**
- 後端合約見第 6 節與 `dmaic-worker.ts` / `server.ts`。

## 4. 八因子魚骨 + 信心度 + FMEA 模型

- **八因子**(6M + 運輸儲存 + 臨床):人/機/料/法/環/運輸與儲存/測量/臨床。選項已針對**抗結痂導尿管**主題客製(塗層製程、尿液 pH 誘發 struvite/apatite 結晶、Proteus 產 urease、留置照護等)。
- **信心度** `mockConfidence()`:由「實質勾選數 + 補充說明 + 是否有支持證據(多來源加權)」決定 score/level。純規則、可重現。
- **根因生成** `buildRootCauses()`:每個有實質勾選的因子生成一筆,帶 `support[] / missing[] / recommend / hypothesis`。
- **待釐清(pending)因子的關鍵處理:** 只勾「待調查」而無實質證據的因子,**不列為已確認根因**,而是標記 `pending:true`、level 降級、statement 標「(待釐清)」。這是為了嚴守 No Hallucination——未證實的因子不得進入「最可能根因」。
- **FMEA/RPN** `mockFMEA()`:
  - 嚴重度 S:阻塞導致臨床傷害 → 高。
  - 發生度 O:pending → 低(3);高信心 → 7。
  - 偵測度 D:缺量測/證據 → 難偵測 → D 高(8);證據充分 → 低(4)。
  - **RPN = S×O×D 決定驗證優先序**;可拖動微調。
- **魚骨自動排版**(移植自舊版並強化):動態字體、車道分配避免同側重疊、文字換行寬度上限 = 相鄰 joint 最小間距 × 0.78,避免標籤壓到下一主幹。低風險因子仍畫出(灰、半透明)供稽核追溯。

## 5. 版面與視覺設計

### 布局
- **固定左側進度列**(`.sidenav`,`--nav-w:200px`)+ 主內容區(`.main`,`max-width` 收斂閱讀寬度)。進度列即八階段導覽,`buildNav()` 動態生成,`done/active` 狀態反映流程位置。
- 每階段一張 `.card`,`module-tag` 標「階段 N」。手機(`max-width:820px`)側欄收起、單欄化。
- 元件即狀態:`pill`(模式徽章)、`level-badge`(信心度)、風險條、FMEA 表 — 用**形狀 + 顏色**讓「該注意的」一眼可辨(UI 是拿來操作/掃視,不是逐字讀)。

### 色彩(語意化,非裝飾)
CSS 變數集中在 `:root`:
- 綠 `--green #1D9E75` = 高信心 / 完成 / 正常。
- 琥珀 `--amber #BA7517` = 中信心 / 待釐清 / 模擬模式(刻意用「警示但非錯誤」色標 pending 與 simulation)。
- 灰 `--gray` = 低風險 / 不相關。
- 藍 `--blue` = 資訊 / AI。
- 紅 `--red` = 高風險 / 錯誤。
- 語意色與品牌強調色分離;good/warning/critical 各有專用色,不與主色混用。

### 寫作策略(copy)
- **繁體中文、資深醫療器材品質/法規工程口吻**;主動語態;控制項寫清楚按下去會發生什麼。
- 提示文字(`.hint`)解釋「為什麼要做這一步」,不只是操作說明。
- 免責與範圍(header 下方)明確寫出:510k/MAUDE/ISO 為**示範資料**;每個結論標信心度與證據來源;資訊不足標「待釐清」而非臆測;正式使用需 QA/RA 審閱。**這段文字是合規要件,不可刪。**

## 6. 後端合約(`/api/dmaic`)

前端 `aiCall(task, context, opts)` → `POST /api/dmaic`,body `{task, context}`,回 `{success, text}`。

| task | 輸入 context | 輸出 | opts.json |
|---|---|---|---|
| `ping` | — | `{success:true,text:"ok"}` | — |
| `summary` | device + 問卷 + AI 調查發現 | 150–220 字問題釐清與收斂摘要 | 否 |
| `rootcause` | 初步根因清單(勾選/證據/信心度) | JSON 陣列 `[{id,rationale,gap}]`(只加敘述) | 是 |
| `report` | 根因(含 RPN 與 pending)+ 驗證計畫 | 150–220 字可追溯結論段落 | 否 |

系統提示(兩份後端一致):資深醫療器材品質工程師(ISO 13485 / CAPA / 失效分析 / V&V / 法規)、繁中、精確可執行、**嚴禁編造**、所有推論須與提供資料一致。

兩種後端實作:
- `server.ts` — 現有 Express + Gemini(repo 沿用)。
- `dmaic-worker.ts` — Cloudflare Worker + Claude Fable 5(符合規格「本專案為 Fable 5 設計」)。合約完全相同,可擇一部署;未部署則前端維持 🟡 模擬模式,功能不減。

## 7. 修改時的護欄(給後續維護者 / AI)

1. **不要把信心度、RPN、因子結構交給模型算** — 那是可稽核的量化基礎,必須留在規則引擎。模型只加工程敘述。
2. **不要移除離線 fallback** — 模型不可用時流程必須照跑。
3. **不要讓 pending 因子進入「已確認根因」** — 違反 No Hallucination。
4. **每個新產出都要能追溯到 `STATE` 的來源欄位** — 否則違反 Traceability。
5. **改架構優先於疊功能**(Refactoring First);單檔已大,新功能先想能否用既有 `STATE`/元件達成。
6. 免責/範圍聲明與「示範資料」標示不可刪。
