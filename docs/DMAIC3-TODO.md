# DMAIC3 後續請 AI 修改的內容(任務清單)

> 這是給下一輪 AI(或人)接手 `DMAIC3.html` 的工作清單。
> **先讀 `docs/DMAIC3-DESIGN.md` 的第 7 節護欄再動手。** 每項標了優先級、動機(降什麼風險/補什麼缺口)、涉及檔案與驗收標準。
> 原則:改架構優先於疊功能;每個結論要能追溯 `STATE`;不知道就標「待釐清」。

圖例:🔴 高 / 🟡 中 / 🟢 低優先。

---

## A. 真實證據整合(目前是示範資料 → 換真實來源)

- 🔴 **A1. 外部知識庫接真實 API**
  - 現況:510k / MAUDE / ISO 為 header 已聲明的**示範資料**。
  - 目標:階段 4 證據彙整改接真實查詢 — FDA 510(k)、MAUDE、PubMed 文獻、ClinicalTrials。
  - 動機:規格要求「根因由多個證據來源推論,絕不只靠單一來源」;示範資料無法真正降風險。
  - 建議:由後端(`dmaic-worker.ts` / `server.ts`)代理查詢,前端仍走 `/api/dmaic` 合約新增 `task:'evidence'`;查不到就明確標「無命中」,**不得編造命中**。
  - 驗收:每筆證據可點回原始來源(510k 號 / MAUDE report id / PMID);查無時顯示缺口而非假資料。

- 🟡 **A2. 證據 ↔ 因子自動對應的可信度標記**
  - 每筆證據標來源類型(內部/510k/MAUDE/文獻)與對應強度,供信心度加權時有稽核軌跡。

## B. AI 互動調查強化(階段 3)

- 🔴 **B1. 動態追問真正由對話收斂**
  - 現況:追問選項與因子預選為規則引擎(保證可追溯)。
  - 目標:連線模式下,讓模型依前一輪回答**動態生成下一題**,但仍把每個回答標記到魚骨因子並寫入 `STATE.investigation.factorHints`。
  - 護欄:模型只影響「問哪些題、收斂到哪些因子方向」;**信心度分數仍由規則引擎算**(見設計文件 §7.1)。
  - 驗收:對話中獲得的每條資訊,最後都出現在根因分析/風險/報告至少一處(規格硬性要求)。

- 🟡 **B2. 問卷分支邏輯擴充**
  - 目前分支式選項較淺;依 invType(失效/客訴/CAPA/NCR/製程異常)展開不同題組。答案未影響後續者不應被問(規格:不問無用問題)。

## C. 報告與匯出(階段 8)

- 🟡 **C1. 匯出可下載報告檔(非只有畫面 + JSON)**
  - 產生 DOCX/PDF:每個結論附信心度與證據來源,每個驗證項附假設↔失效↔風險對應。
  - 動機:交付物要能進公司文件系統。舊 `DMAIC.html` 有 docx.umd.js 產 CAPA 報告的邏輯可參考移植。
  - 驗收:報告可追溯、含免責、版本/日期,RA 可審閱。

- 🟢 **C2. 報告加入「證據缺口彙總表」**
  - 把所有 `missing[]` 集中成一張表 + 對應驗證項,讓「結案前要補什麼」一目了然。

## D. 風險與驗證(階段 6–7)

- 🟡 **D1. FMEA 預設值說明化**
  - 現況 S/O/D 由信心度/證據規則推估。加上每個預設值的一句理由(為何 O=7、為何 D=8),提升可稽核性。

- 🟡 **D2. 驗證方法庫對應統計/工程方法 + ISO 條款**
  - 每個驗證項對應合適方法(SPC / MSA / DOE / 假設檢定 / 生物相容 ISO 10993 / 製程確效 §7.5.6 等)。ISO 條款須標「AI 建議,RA 確認」,不得當定論。

## E. 後端 / 部署

- 🔴 **E1. 決定正式後端模型與部署**
  - 兩份後端已備:`server.ts`(Gemini)/ `dmaic-worker.ts`(Claude Fable 5)。擇一設定金鑰並部署,DMAIC3 即進 🟢 連線模式。
  - Worker 部署後,若前端與 Worker 不同網域,需將 `DMAIC3.html` 的 `AI.base` 指向 Worker URL 並確認 CORS。
  - 注意:Fable 5 需 30 天資料保留(非 ZDR);Worker 已內建 refusal 處理與 server-side fallback 到 Opus 4.8。

- 🟢 **E2. Worker 加上 rate limit 與簡單快取**
  - 相同 context 的 summary/report 可短期快取,降成本降延遲。

## F. 品質 / 可維護性

- 🟢 **F1. 單檔過大 → 考慮拆分**
  - `DMAIC3.html` 已含大量 CSS/JS。若持續加功能,評估拆成模組(但須保留「單檔可離線開啟」這個對使用者很方便的特性 → 拆分需搭配建置產生單檔版)。

- 🟢 **F2. 自我檢查清單**
  - 每次修改後跑一遍八階段 headless 煙霧測試(離線 + 連線兩模式),確認 0 JS 錯誤(這是既有驗證標準)。

## 給 AI 的一句話交接

> 你要修改的是一套**醫療器材失效調查輔助系統**,目標是**降低產品風險**,不是把報告寫漂亮。
> 信心度/風險/因子結構屬於可稽核的量化基礎,留在規則引擎;模型只加工程敘述。
> 不知道就標「待釐清」,每個結論都要能追溯到 `STATE` 的來源。改動前先讀 `docs/DMAIC3-DESIGN.md`。
